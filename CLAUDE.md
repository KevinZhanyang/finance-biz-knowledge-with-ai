# 仓库规范

## 定位

中国金融业务知识库。记录行业的业务运转逻辑、机构协作机制和领域术语。

## 内容原则

- **重业务逻辑**：记录"行业怎么运转的"，不写金融理论或投资策略
- **重流程对接**：关注机构间如何协作、数据如何流转
- **实操导向**：结合实际业务场景，附流程图和示例
- **术语先行**：每个领域先梳理核心术语，再展开业务流程
- **准确性**：涉及监管规则、费率、时间节点等须标注信息来源或时效

## 目录结构

```
fund/           基金行业（销售、登记结算、估值、托管）
institution/    行业参与机构（基金公司、代销机构、托管行、中登、证监会、交易所）
research/       投研体系（投研流程、资产配置、风控、合规）
insurance/      保险行业（产品、渠道、运营）
integration/    系统对接（接口协议、数据格式、报文流转）
glossary/       术语表（基金术语、交易结算术语、通用术语）
```

新增模块时在根目录创建文件夹，并更新 README.md 的目录结构。

## 文档格式规范

### 文件命名
- 使用小写英文 + 短横线：`day-end.md`、`fund-terms.md`
- 文件名应能直观反映内容

### 文档结构
- 一级标题（`#`）：文档主题，每个文件仅一个
- 二级标题（`##`）：主要章节
- 三级标题（`###`）：细分内容
- 待补充的文件以 `> TODO: 待补充` 开头，保留要点提纲

### 内容格式
- 用**表格**整理对比性信息（术语表、分类对比、费率对比）
- 用 **ASCII 流程图**（代码块内）展示业务流转
- 用**时间线**展示有序流程
- 用**列表**罗列要点，避免大段落
- 文档间用**相对路径**互相引用：`[日终处理](../registration/day-end.md)`

### 术语表规范（glossary/）
- 使用 Markdown 表格，列：术语 | 英文（可选）| 含义
- 按主题分组（交易相关、净值相关、费率相关等）
- 术语在其他文档首次出现时加粗，可链接到术语表

## 协作流程

1. 确定要补充的模块/文件
2. 如果是新领域，先补充对应的术语表
3. 编写正文内容，遵循格式规范
4. 检查与已有文档的交叉引用
5. 更新 TODO.md 状态
