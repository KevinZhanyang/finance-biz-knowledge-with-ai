# TA 系统（Transfer Agent）

## 什么是 TA

TA（Transfer Agent，转让代理/注册登记）系统是基金行业的**核心枢纽**，负责记录"谁持有多少份额"。类比银行的核心账务系统记录"谁有多少钱"，TA 系统记录"谁有多少基金份额"。

**一句话定义**：TA 系统是基金份额的"户口本"，所有份额的产生、变动、消亡都必须经过 TA 登记才算数。

## TA 的核心职能

```
                        TA 系统
                    ┌─────────────┐
  投资者开户  ───→  │  账户管理    │
  申购/赎回   ───→  │  交易确认    │  ───→  份额确认回传给销售渠道
  基金分红    ───→  │  份额登记    │  ───→  清算数据给托管银行
  转换/转托管 ───→  │  权益处理    │  ───→  对账文件给各方
                    └─────────────┘
```

### 1. 账户管理

- **基金账户开户**：为投资者建立基金账户，分配唯一的基金账号
- **账户信息维护**：姓名、证件、银行卡、风险等级等
- **账户冻结/解冻**：司法冻结、质押等场景
- **一人多卡**：一个基金账户可关联多个交易账户（不同销售渠道）

#### 账户体系

```
投资者（身份证号）
  └── 基金账户（TA 分配，全局唯一）
        ├── 交易账户 A（在天天基金开立）
        ├── 交易账户 B（在招商银行开立）
        └── 交易账户 C（在基金公司直销开立）
```

- **基金账户**：由 TA 系统管理，一个投资者在一家基金公司只有一个基金账户
- **交易账户**：由销售机构管理，投资者在每个销售渠道各有一个交易账户
- 一个基金账户下可挂多个交易账户，份额归属于基金账户层面

### 2. 交易确认

这是 TA 最核心的日常工作，在[日终跑批](./day-end.md)中完成：

```
各销售渠道提交交易申请（T日白天）
    ↓
TA 汇总所有申请
    ↓
获取 T 日确认净值
    ↓
逐笔计算：
  申购 → 计算确认份额
  赎回 → 计算赎回金额
  转换 → 计算转出金额和转入份额
    ↓
更新份额余额
    ↓
生成确认结果文件 → 回传给各销售渠道
```

### 3. 份额登记

TA 系统维护**权威的份额登记簿**：

| 字段 | 说明 |
|------|------|
| 基金账号 | 投资者标识 |
| 基金代码 | 哪只基金 |
| 总份额 | 当前持有的总份额 |
| 可用份额 | 可交易的份额（总份额 - 冻结份额） |
| 冻结份额 | 因赎回在途、司法冻结等不可交易的份额 |
| 交易账号 | 通过哪个销售渠道持有 |

### 4. 权益处理

- **分红处理**：根据权益登记日的份额，计算每个持有人应得分红
- **基金拆分/合并**：调整份额数量和净值，总资产不变
- **基金清盘**：计算每个持有人的清算款

## 自建 TA vs 中登 TA

基金行业有两套 TA 体系并行运作：

|  | 基金公司自建 TA | 中登 TA（中国结算） |
|--|---------------|-----------------|
| 适用范围 | 场外基金（通过银行、第三方代销） | 场内基金（ETF、LOF 的场内交易部分） |
| 运营方 | 各基金公司自行运营或外包 | 中国证券登记结算公司统一运营 |
| 账户体系 | 基金账户 | 证券账户（股东账户） |
| 清算方式 | 基金公司与各代销机构逐一清算 | 通过中登统一净额清算 |
| 交易确认 | T+1 确认 | T+1 确认 |
| 典型场景 | 投资者在天天基金买了一只混合基金 | 投资者在券商账户买了一只 ETF |

### 自建 TA 的主流方案

大多数基金公司不会从零开发 TA 系统，而是采购行业解决方案：

| 厂商 | 产品 | 市场地位 |
|------|------|---------|
| 恒生电子 | TA 5.0 系列 | 市占率最高，公募基金行业标配 |
| 赢时胜 | TA 系统 | 部分中小基金公司使用 |
| 金证股份 | TA 系统 | 部分基金公司使用 |

### 跨 TA 场景

一只基金可能同时涉及两套 TA：

```
场外申购（银行代销）→ 基金公司自建 TA 登记
场内买入（券商交易）→ 中登 TA 登记

同一只 LOF 基金，场内场外份额分别登记在不同 TA 中
投资者可通过"转托管"将份额在场内场外之间转移
```

## TA 与各方的数据交互

### 交互关系全景

```
              ┌──────────┐
              │  托管银行  │
              └────┬─────┘
                   │ 清算数据、资金对账
                   ↕
┌──────────┐    ┌──────┐    ┌──────────┐
│ 代销机构A │←→│  TA  │←→│ 代销机构B │
└──────────┘    └──┬───┘    └──────────┘
                   │
                   ↕
              ┌──────────┐
              │ 直销渠道  │
              └──────────┘
```

### 与代销机构

| 方向 | 数据内容 | 时间 |
|------|---------|------|
| 代销 → TA | 交易申请（申购、赎回、转换等） | T 日 15:00 后汇总发送 |
| TA → 代销 | 交易确认结果 | T 日日终跑批后 |
| TA → 代销 | 份额对账文件 | T+1 日 |
| 代销 → TA | 开户/账户变更申请 | 实时或批量 |

### 与托管银行

| 方向 | 数据内容 | 时间 |
|------|---------|------|
| TA → 托管 | 清算数据（各渠道应收应付） | T 日日终后 |
| TA → 托管 | 基金份额变动汇总 | T+1 日 |
| 托管 → TA | 资金到账确认 | T+1 日 |

### 数据交换方式

TA 与外部机构的数据交换主要通过**接口文件**（批量文件交换）：

- **文件格式**：TXT 定长格式，遵循行业协议（Open 协议 / 中登协议）
- **传输通道**：专线、深证通、VPN 加密通道
- **交换频率**：日终批量交换为主，部分支持日间实时交互
- **核心文件类型**：

| 文件 | 方向 | 内容 |
|------|------|------|
| 交易申请文件 | 代销→TA | 当日所有申购/赎回/转换申请 |
| 交易确认文件 | TA→代销 | 确认结果（份额/金额/费用） |
| 份额对账文件 | TA→代销 | 投资者份额余额快照 |
| 分红明细文件 | TA→代销 | 每个持有人的分红金额 |
| 清算汇总文件 | TA→托管 | 各渠道应收应付汇总 |

详见 [行业接口协议](../../integration/protocols.md) 和 [数据文件格式](../../integration/data-formats.md)

## TA 系统的关键特性

### 数据权威性

- TA 的份额登记簿是**法律认可的权威记录**
- 当代销机构的数据与 TA 不一致时，以 TA 为准
- 对账发现差异时，代销机构需根据 TA 数据进行调整

### 高可靠性要求

- TA 日终跑批不能失败，否则影响全行业链路（见[日终处理](./day-end.md)）
- 份额数据零容错：多一分或少一分都是事故
- 金额计算精度要求：通常保留小数点后 2 位（份额保留后 2~4 位）

### 业务规则引擎

TA 内置了大量业务规则：
- 申购费率分档（按金额区间适用不同费率）
- 赎回费率分档（按持有天数适用不同费率）
- 巨额赎回判断与处理
- 大额申购限制
- 最低持有份额检查
- 封闭期/限制期控制
- 分红方式默认值及修改规则

## TA 在业务链路中的位置

```
投资者 → 销售机构 → TA 系统 → 托管银行
           ↑           ↑          ↑
         客户入口    业务中枢    资金保管
         交易受理    份额登记    资金清算
         客户服务    交易确认    净值复核
```

TA 是连接**销售端**和**资金端**的中枢：
- 向左（销售端）：接收交易申请、回传确认结果
- 向右（资金端）：提供清算数据、驱动资金划付
- 没有 TA 的确认，份额不会变动，资金不会划转
